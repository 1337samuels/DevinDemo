How It Runs

⚡

On-Demand

Trigger from the Dashboard or CLI for any repo, any time. Full pipeline or individual phases.

🕓

Scheduled

GitHub Actions runs the full pipeline every week automatically. Tech debt is surfaced continuously — no manual intervention.

Phase 1: Scan — Fast Candidate Detection

⚙
Feature Flags + Staleness
env vars, config gates & stale flags
Detects env var checks, boolean configs, and gate functions. Prioritizes stale flags: hardcoded constants with no dynamic override, always-true branches, and names referencing old versions/dates.
🚫
Unreachable Code
dead branches & guards
Unreachable branches (if False, if 0), always-true guards, and code paths that can never execute.
🗑
Unused Definitions
strong signals only
Functions/classes with strong unused signals: private (prefixed with _) and never referenced, or in modules never imported. Avoids flagging framework handlers, CLI commands, and test fixtures — Phase 2 handles full reachability.
📝
Commented-Out Code
≥3 lines of former source
Large blocks of commented-out code (3+ consecutive lines) that look like former source code. Phase 2 verifies these are actual dead code, not documentation or examples.
🔨
Tech Debt (Actionable)
stale TODOs & dated markers
Prioritizes TODOs with staleness signals: ticket references, past version numbers, dates, or phrases like “remove after” / “temporary workaround.” Skips generic TODOs with no actionable context.
⚠
Deprecated APIs
stdlib & third-party
Usage of deprecated standard library or third-party APIs (e.g. optparse, imp) that should be replaced with modern alternatives.
🔄
Compat Shims
old Python versions
Compatibility shims for old Python versions, sys.version checks, six library usage, and try/except import patterns for merged stdlib modules.

Phase 2: Validate — 8 Verification Layers

🔎
Re-Confirm (Category-Aware)
Layer 1
Category-aware validation: Feature flags → verify actual flag pattern, not just a matching name. Commented-out code → confirm it's former source code (not docs/examples). Tech debt → confirm the marker still exists and is actionable. Only EXEMPTs truly invalid detections.
⏰
Git History Staleness
Layer 2
Check when the code was last meaningfully edited. Filter out bulk/cosmetic commits. Flag as stale if untouched for >365 days.
👥
Active Development
Layer 3
Search for open PRs or recent branches that touch this code. If someone is actively working on it, lower the removal confidence.
🔗
Static Reachability
Layer 4
Trace call chains from entry points. Check if any execution path can reach this code. Detect framework patterns that may hide usage.
🎯
Issue Archaeology
Layer 5
Search issues, PR comments, and commit messages for discussions about this code. Determine community sentiment toward removal.
📚
Test Coverage
Layer 6
Check if any tests directly reference or exercise this code. Identify test files that would break if the candidate is removed.
🚀
Runtime Signals
Layer 7
Look for flag evaluation counts, APM invocation data, and references in deployment/infra configs that indicate runtime usage.
🛠
External Consumers
Layer 8
Check if the symbol is exported, part of a published package, or consumed by external repos/services. Flag API endpoints.

Phase 3: Cleanup — PR Generation

🚀
High Confidence
auto-generate PRs
Issues with HIGH confidence automatically get a cleanup PR generated via Devin API, scoped to a single candidate.
🔀
Scoped PRs
one candidate per PR
Each PR addresses exactly one candidate with a clear title, description, and context for the reviewer.
✅
Auto-Merge
all 8 layers passed
Issues that passed all 8 verification layers can be auto-merged. optional — auto-merge
👀
Manual Review
low confidence items
Low-confidence issues are flagged for manual review only — no PR is generated, but findings are reported for human evaluation.

Phase 4: Report — Publish Results

📝
Notion Table
full findings DB
Creates a Notion database with all findings, confidence levels, 8 layer checkboxes, PR links, code snippets, and detection reasoning.
💬
Slack Alerts
PR notifications
Sends Block Kit messages to a Slack channel with PR link, last commit date, and all legacy reasons when a PR is opened.
📄
GitHub PRs
cleanup pull requests
Links to all auto-generated cleanup PRs from Phase 3, with status tracking (open, merged, closed).
💾
JSON Artifacts
machine-readable output
Structured JSON output at every phase (scan.json, validate.json, cleanup.json) for CI/CD integration and downstream tooling.
